# scheelite ‚Äî `/` and `/nix` legacy-mountpoint migration

Status: active (scaffold, pre-investigation)
Owner: dan
Last updated: 2026-06-01

## Why this plan exists

The May 2026 legacy-mountpoint migration
(`docs/plans/completed/scheelite-zfs-legacy-retry.md`) flipped 40 of
the 42 ZFS datasets on scheelite from path mountpoints to
`mountpoint=legacy`. The remaining two ‚Äî `scheelite-root/local/root`
(`/`) and `scheelite-root/local/nix` (`/nix`) ‚Äî were left as path
mountpoints because they're held by every running process and the
prior plan triaged them out of scope.

This plan covers the flip. The lesson from May's `/persist` and
`/home` work is that `zfs set -u` from the running system is the
correct tool for can't-unmount datasets; rescue USB is the rollback
escape hatch, not the primary procedure.

## Open question gating the whole plan (do this first)

**Has the dual-mount race actually been observed on `/` or `/nix`
post-migration?** The structural risk is real but the practical
impact is unknown. If no race has been logged, this is hygiene rather
than firefighting and the urgency drops.

```fish
sudo journalctl -b 0 -u zfs-mount.service --no-pager
sudo journalctl --since '14 days ago' --no-pager \
  | grep -E '(zfs_mount_at|already mounted|directory not empty).*/(nix|sysroot|/)\b' \
  | head -40
```

If nothing in 14 days: park this plan, revisit on next NixOS bump.

## Findings

### F1 ‚Äî Use `zfs set -u` from the running system, not rescue USB

`/` and `/nix` are held open by every process; they can't be
unmounted live. `zfs set -u <prop>=<val>` sets the property without
attempting unmount/remount ‚Äî this is exactly the case the flag is
designed for, and it's what F6 of the prior plan used for `/persist`
and `/home`. Rescue USB is retained as the rollback path only (F10
analog).

### F2 ‚Äî Two separate `nix eval` gates before deploy

Userspace fstab and initrd's stage-1 mount config are generated by
different code paths. Verify both:

```fish
# Userspace fstab options
nix eval --json '.#nixosConfigurations.scheelite.config.fileSystems."/".options'
nix eval --json '.#nixosConfigurations.scheelite.config.fileSystems."/nix".options'
# Initrd's mount config
nix eval --json '.#nixosConfigurations.scheelite.config.boot.initrd.systemd.fileSystems."/".options' 2>&1
nix eval --json '.#nixosConfigurations.scheelite.config.boot.initrd.systemd.fileSystems."/nix".options' 2>&1
```

**Acceptance**: all four return `[]` (no `zfsutil`).

If any returns `["zfsutil"]` ‚Äî stop. Investigate whether disko's
gating applies to the initrd path, then revisit.

### F3 ‚Äî Disko config change

`nixos-configurations/scheelite/disko.nix`: change `local/root` and
`local/nix`'s `options.mountpoint` from path values to `"legacy"`.
Single commit. Same shape as commit `58ae74e`'s 40-dataset change.

### F4 ‚Äî Garbage-collect pre-migration generations

Older generations have fstab entries with `zfsutil` for `/` and
`/nix`. After the live property flip, those generations are
unbootable ‚Äî `mount.zfs zfsutil` against a `legacy` dataset is
refused. Per the prior plan's F9, remove them before the operator
has a chance to pick the wrong one at the systemd-boot menu:

```fish
sudo nix-collect-garbage --delete-older-than 1d
sudo /run/current-system/bin/switch-to-configuration boot
```

### F5 ‚Äî Rollback-root initrd service interaction

`nixos-configurations/scheelite/default.nix:34-45` runs `zfs rollback
-r zroot/local/root@empty` before `sysroot.mount`. Verify the unit
name and ordering still hold after the flip:

```fish
nix eval --raw '.#nixosConfigurations.scheelite.config.boot.initrd.systemd.services.rollback-root.before'
nix eval --raw '.#nixosConfigurations.scheelite.config.boot.initrd.systemd.services.rollback-root.after'
```

**Acceptance**: still references `sysroot.mount` and
`zfs-import-zroot.service` respectively (or whatever the new pool's
import unit is named). Mountpoint property is not snapshot-recorded,
so the `@empty` snapshot target remains valid.

### F6 ‚Äî Live migration procedure

```fish
# Deploy new generation without activating
sudo nixos-rebuild boot --flake .#scheelite

# Flip the two live properties without unmount
sudo zfs set -u mountpoint=legacy scheelite-root/local/root
sudo zfs set -u mountpoint=legacy scheelite-root/local/nix

# Verify pool state
zfs get -H mountpoint scheelite-root/local/root scheelite-root/local/nix

# Reboot ‚Äî new generation activates from scratch with new fstab
sudo systemctl reboot
```

### F7 ‚Äî Higher reboot bar for the rootfs case

The prior plan's "three consecutive reboots" bar verified data
datasets. For rootfs, also exercise an activation cycle to confirm
re-mounts work end-to-end:

- 3√ó `sudo systemctl reboot` ‚Äî verify clean boot each time.
- 1√ó `sudo nixos-rebuild switch --flake .#scheelite` (no-op rebuild)
  ‚Äî confirm activation doesn't choke on the new mount setup.
- `journalctl -b 0 -u zfs-mount.service` ‚Äî should report no
  `/sysroot` or `/sysroot/nix` mount attempts.

### F8 ‚Äî Rollback path (only if reboot fails)

Same procedure as the prior plan's F10. Pre-emptively: do NOT pick an
older generation from the systemd-boot menu if the new generation
fails. Boot the NixOS installer USB:

```bash
zpool import -f -N -R /mnt scheelite-root
zfs set -u mountpoint=/    scheelite-root/local/root
zfs set -u mountpoint=/nix scheelite-root/local/nix
zpool export scheelite-root
reboot
```

## Exit criteria

- Both `nix eval` gates from F2 return `[]`.
- F4 garbage-collect step run.
- F6 live migration completes without error.
- F7 reboot + activation bar met.
- `zfs-mount.service` journal shows no attempts against `/` or `/nix`
  (or shows them no-op'd cleanly).
- No `directory not empty` or `zfs_mount_at` errors on host mount
  units.

## What this plan does *not* do

- Does not flip `boot.zfs.forceImportRoot` (separate plan).
- Does not change `tank0` datasets (already flipped in commit
  `58ae74e`).
- Does not address `local` parent (`canmount=off`, no mountpoint
  property).
- Does not deploy to other hosts (argentite, cassiterite, malachite ‚Äî
  only scheelite uses impermanence; the others don't have the same
  legacy-mountpoint structure).

## Stop conditions

- Either `nix eval` gate from F2 returns `["zfsutil"]`.
- F5 verification shows rollback-root ordering broken by the flip.
- Any reboot in F7 fails or surfaces a new `zfs_mount_at` error.

In any of these cases, do not proceed (or roll back via F8).
