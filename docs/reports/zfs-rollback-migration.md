# ZFS Rollback Migration: `boot.initrd.postDeviceCommands` to `boot.initrd.systemd.services`

**Date:** 2026-04-25
**Machines affected:** scheelite (fixed), malachite, argentite, cassiterite (pending)
**Status:** Partially complete (scheelite only)

## Problem

Running `nix flake show` produced the following assertion failure:

```
Failed assertions:
- systemd stage 1 does not support `boot.initrd.postDeviceCommands`. Instead, create systemd services using the `boot.initrd.systemd.services` options, which has an API matching the stage 2 `systemd.services` options. Refer to `bootup(7)`, specifically the sections on "Bootup in the Initrd" and "System Manager Bootup", for information about when various units happen, and order services accordingly.

    Definitions:
    - /nix/store/75bkaivfbwq3x8cs7155hag7hs1chjcx-source/flake.nix
```

### Root cause

NixOS unstable (`nixpkgs-unstable`) has made `boot.initrd.systemd.enable = true` the default. This means the initrd now uses systemd as its init system (referred to as "systemd stage 1"). The legacy scripted initrd option `boot.initrd.postDeviceCommands` is explicitly rejected when systemd stage 1 is active. The NixOS module at `nixos/modules/system/boot/systemd/initrd.nix` contains an assertion that checks for this and produces the error message above.

### The old code

In `nixos-configurations/scheelite/default.nix`, the ZFS rollback for impermanence was implemented using the legacy scripted initrd hook:

```nix
boot.initrd.postDeviceCommands = lib.mkAfter ''
  zfs rollback -r scheelite-root/local/root@empty
'';
```

This ran the ZFS rollback after device nodes were available but before the root filesystem was mounted. It worked with the scripted initrd but is not compatible with systemd stage 1.

## Solution

The fix replaces `boot.initrd.postDeviceCommands` with an equivalent systemd service defined via `boot.initrd.systemd.services`:

```nix
boot.initrd.systemd.services.rollback-root = {
  description = "Rollback ZFS root to empty snapshot";
  wantedBy = [ "initrd.target" ];
  after = [ "zfs-import-scheelite-root.service" ];
  before = [ "sysroot.mount" ];
  unitConfig.DefaultDependencies = "no";
  serviceConfig.Type = "oneshot";
  path = [ config.boot.zfs.package ];
  script = ''
    zfs rollback -r scheelite-root/local/root@empty && echo "rollback of scheelite-root complete"
  '';
};
```

### Explanation of each attribute

#### `description`

Human-readable description of what the service does. Shows up in `systemctl status` and journal logs.

#### `wantedBy = [ "initrd.target" ]`

This ensures the service is pulled into the initrd boot sequence. Without this, the service would be defined but never started. `initrd.target` is the main target for the initrd phase. The NixOS ZFS module itself uses `initrd.target` for its own initrd targets (`targets.zfs.wantedBy = [ "initrd.target" ]`).

#### `after = [ "zfs-import-scheelite-root.service" ]`

The rollback must happen after the ZFS pool is imported, otherwise the datasets are not accessible and `zfs rollback` would fail.

The service name `zfs-import-scheelite-root.service` is auto-generated by the NixOS ZFS module. The module creates import services using the pattern `zfs-import-<poolname>` where the pool name is extracted from the first path component of ZFS `fileSystems` device paths. In scheelite's `hardware.nix`, the root filesystem is:

```nix
fileSystems."/" = {
  device = "scheelite-root/local/root";
  fsType = "zfs";
  options = [ "zfsutil" ];
};
```

The pool name `scheelite-root` is extracted from `scheelite-root/local/root` (everything before the first `/`), producing the service name `zfs-import-scheelite-root.service`.

This is defined in the NixOS ZFS module at `nixos/modules/tasks/filesystems/zfs.nix` in the `createImportService` function, which creates services via:

```nix
lib.nameValuePair "zfs-import-${pool}" { ... }
```

The `rootPools` variable is computed from all ZFS filesystems that are `neededForBoot`:

```nix
rootPools = lib.unique (map fsToPool (lib.filter utils.fsNeededForBoot zfsFilesystems));
```

**Important caveat:** Getting the pool name wrong is a common mistake. A NixOS Discourse thread documented a case where a user used `zfs-import-zroot.service` when their pool was actually named `rpool`, causing silent failure. Always verify the pool name matches your `fileSystems` device paths.

#### `before = [ "sysroot.mount" ]`

The rollback must happen before the root filesystem is mounted. In the systemd initrd, the root filesystem is mounted at `/sysroot` (not `/`), and the corresponding systemd mount unit is `sysroot.mount`. This unit is auto-generated by `systemd-fstab-generator` from the initrd fstab entries that NixOS creates.

The ZFS import service itself imports the pool with the `-N` flag (no auto-mount), meaning it imports the pool but does not mount any datasets. Mounting is handled separately by systemd mount units generated from `fileSystems`. The import service uses `prefix = "/sysroot"` for mount paths in the initrd.

The combined ordering is: pool import -> rollback -> root mount.

#### `unitConfig.DefaultDependencies = "no"`

By default, systemd adds implicit dependencies on `sysinit.target`, `basic.target`, and `shutdown.target` to all services. In the initrd environment, these default dependencies can interfere with early boot ordering. They might cause the rollback service to be ordered too late (after `sysroot.mount` has already run) or create circular dependency chains.

Setting `DefaultDependencies = "no"` disables these implicit dependencies, giving full manual control over ordering through the explicit `after`, `before`, and `wantedBy` directives.

#### `serviceConfig.Type = "oneshot"`

The service runs a single command and exits. `oneshot` tells systemd to consider the service "active" only while the script is running and to wait for it to complete before starting units that are ordered after it.

#### `path = [ config.boot.zfs.package ]`

Puts the ZFS package on the service's `$PATH` so that `zfs` can be called directly in the script without a full store path.

Using `config.boot.zfs.package` rather than `pkgs.zfs` is important because it respects any overrides to `boot.zfs.package` (e.g., if someone sets it to `pkgs.zfs_unstable`). The NixOS ZFS module internally uses `config.boot.zfs.package` (aliased as `cfgZfs.package`) for the same reason.

An alternative approach is to interpolate the full path in the script:

```nix
script = ''
  ${config.boot.zfs.package}/bin/zfs rollback -r scheelite-root/local/root@empty
'';
```

Both work. The `path` approach was chosen for readability. Note that `/bin/zfs` and `/sbin/zfs` in the ZFS package are hard links to the same binary, so either path works.

#### `script`

The actual rollback command. `zfs rollback -r scheelite-root/local/root@empty` rolls back the `scheelite-root/local/root` dataset to the `empty` snapshot, including recursively destroying any snapshots created after `empty`. This is the core of the impermanence pattern: every boot starts from a clean root.

The `&& echo "rollback of scheelite-root complete"` provides a confirmation message in the journal on success, which is helpful for debugging boot issues.

## Systemd initrd boot sequence context

For reference, the relevant portion of the systemd initrd boot sequence is:

1. `sysinit.target` - basic system initialization
1. Device discovery and udev
1. `systemd-modules-load.service` - kernel modules loaded
1. `zfs-import-scheelite-root.service` - ZFS pool imported (with `-N`, no auto-mount)
1. **`rollback-root.service`** - ZFS rollback happens here
1. `sysroot.mount` - root filesystem mounted at `/sysroot`
1. Other mount units (`sysroot-nix.mount`, `sysroot-home.mount`, `sysroot-persist.mount`, etc.)
1. `initrd-root-fs.target`
1. `initrd-fs.target`
1. `initrd-nixos-activation.service`
1. `initrd-switch-root.target` - switch to real root

The `zfs-import-scheelite-root.service` ordering is:

- **after:** `systemd-modules-load.service`, `systemd-ask-password-console.service`
- **requiredBy:** all mount units for that pool (e.g., `sysroot.mount`, `sysroot-nix.mount`) plus `zfs-import.target`
- **before:** all mount units for that pool, `shutdown.target`, `zfs-import.target`

The target chain is: `zfs-import-<pool>.service` -> `zfs-import.target` -> `zfs.target` -> `initrd.target`

## Verification

The implementation was verified against multiple independent sources:

### NixOS source code

- The ZFS module at `nixos/modules/tasks/filesystems/zfs.nix` confirms the `zfs-import-<poolname>` service naming convention and the `createImportService` function.
- The systemd initrd module at `nixos/modules/system/boot/systemd/initrd.nix` confirms that `boot.initrd.postDeviceCommands` is explicitly rejected via an assertion when `boot.initrd.systemd.enable = true`.
- `boot.initrd.systemd.enable` defaults to `true` in current NixOS unstable.

### Community sources

The following community sources all use the same pattern (`wantedBy = initrd.target`, `after = zfs-import-<pool>.service`, `before = sysroot.mount`, `DefaultDependencies = "no"`, `Type = "oneshot"`):

- [Ephemeral root partition on NixOS using ZFS - notthebe.ee](https://notthebe.ee/blog/nixos-ephemeral-zfs-root/)
- [Encrypted Root and ZFS on NixOS - Ryan Seipp](https://ryanseipp.com/posts/nixos-encrypted-root/)
- [systemd initrd on NixOS - blog.decent.id](https://blog.decent.id/post/nixos-systemd-initrd/)
- [Full Disk Encryption and Impermanence on NixOS - notashelf](https://notashelf.dev/posts/impermanence)

### NixOS Discourse threads

- [ZFS rollback not working using boot.initrd.systemd](https://discourse.nixos.org/t/zfs-rollback-not-working-using-boot-initrd-systemd/37195) - documents a common pitfall of getting the pool name wrong in the `after` directive.

## Outstanding work: other machines

Three other machines in this repository also use ZFS with impermanence but do **not** currently define a rollback initrd service:

| Machine | Pool name | Snapshot | Has rollback service? |
|---|---|---|---|
| scheelite | `scheelite-root` | `scheelite-root/local/root@empty` | Yes (fixed in this session) |
| malachite | `zroot` | `zroot/local/root@empty` | No |
| argentite | `zroot` | `zroot/local/root@empty` | No |
| cassiterite | `zroot` | `zroot/local/root@empty` | No |

Evidence that these machines expect rollback:

- All three set `security.sudo.extraConfig = "Defaults lecture = never"` which is specifically needed when root is rolled back on every boot (sudo lectures would repeat otherwise).
- All three create the `@empty` snapshot in their disko configurations.

For these machines, the rollback service would use `zfs-import-zroot.service` in the `after` directive (since their pool is named `zroot` rather than `scheelite-root`):

```nix
boot.initrd.systemd.services.rollback-root = {
  description = "Rollback ZFS root to empty snapshot";
  wantedBy = [ "initrd.target" ];
  after = [ "zfs-import-zroot.service" ];
  before = [ "sysroot.mount" ];
  unitConfig.DefaultDependencies = "no";
  serviceConfig.Type = "oneshot";
  path = [ config.boot.zfs.package ];
  script = ''
    zfs rollback -r zroot/local/root@empty && echo "rollback of zroot complete"
  '';
};
```

These machines may not currently be failing the assertion if they are not using `boot.initrd.postDeviceCommands`, but they also may not be rolling back their root on boot, which defeats the purpose of the impermanence setup. This should be investigated separately.
